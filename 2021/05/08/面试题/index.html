ArrayList和LinkedList的区别

  • ArrayList底层基于动态数组,使用下标随机访问元素的效率高(就是查询效率高),向数组尾部添加元素的效率高,但是删除数组中间的数据,或者是向数组中间添加数据的效率比较低,因为需要对数组中的元素进行移位,最坏的情况就是,删除数组第一个元素,那么后面的元素都需要向前进一位
  • LinkedList底层基于链表,因此添加删除元素的效率高,只需要改变指针指向即可,但是查询效率低,因为需要遍历链表;
  • (linkedList的删除也分两种情况,一种是知道待删除元素的位置,另一种是不知道待删除元素的位置,前者的情况下,只需要改变指针指向即可,时间复杂度O(1),后者的情况下,需要遍历链表,时间复杂度O(n),比如removeLastOccurrence方法)

HashMap的特性

  1. HashMap存储键值对,允许为null,key值不可重复,若重复则覆盖
  2. 非同步,线程不安全
  3. 底层是hash表,不保证有序

谈一下hashMap的底层原理

  • 基于hashing的原理,jdk8之后采用数组+链表+红黑树的数据结构
  • 通过put和get来存储或获取对象
  • 当我们给put方法传递键值对时,先对key做一个hashCode()的计算来得到它在bucket数组中的位置来存储entry对象;获取对象时,通过get获取到bucket的位置,再通过key的equals方法找到正确的键值对,然后返回value

HashMap的put方法

  • 首先将key通过hash算法得到一个哈希值作为数组下标

  • 如果数组对应下标位置元素为空,则将key和value封装为entry对象(1.8为node对象),并放入该位置

  • 如果数组对应下标位置元素不为空,则分情况讨论

    • 1.8的话,会判断该位置上的node的类型,判断是红黑树还是链表,
      • 如果是红黑树,则会将key和value封装为一个红黑树节点并添加到红黑树中去,在这个过程中,会去判断红黑树中是否存在当前key,如果存在会更新value
      • 如果是链表,则封装为一个entry对象,使用尾插法插入链表,遍历链表,在遍历链表的过程中会判断是否存在当前key,如果存在则更新value,插入完成后,判断当前链表的节点数是否大于等于8,如果大于等于8,则将该链表转为红黑树
      • 将key和value装封装为node插入到链表或者红黑树中后,在判断是否需要进行扩容,需要就扩容,不需要就结束put方法
    • 1.8以前,会先判断是否需要扩容,如果需要扩容就进行扩容,如果不需要就封装为entry对象,使用头插法插入当前链表中
  1. 计算key的hashCode值
  2. 如果散列表为空时,调用resize()初始化散列表
  3. 如果没有发生碰撞,直接添加元素到散列表中去
  4. 如果发生了碰撞(hashCode相同),进行三种判断
    1. 如果key地址相同或则equals后相等,则替换旧值
    2. 如果是红黑树结构,就调用树的插入方法
    3. 链表结构,循环遍历直到链表中某个节点为空,尾插法进行插入,插入后判断链表节点数是否转为红黑树,也可以遍历到有节点与插入元素的哈希值和内容相同，进行覆盖
  5. 如果桶满了大于阀值，则resize进行扩容

  2. 因为这些类是Immutable的,并且这些类已经很规范的覆写了hashCode和equals方法,作为不可变类,天生就是线程安全的,而且可以很好的优化比如缓存hash值,避免重复计算等等

ThreadLocal

  • threadLocal是Java提供的线程本地存储机制,可以使用该机制将数据缓存在线程内部,该线程的所有方法均可获取到缓存的数据

  • threadLocal的底层是通过ThreadLocalMap来实现的,每个Thread对象中都存在一个ThreadLocalMap, map的key为ThreadLocal对象,value为缓存的数据

  • threadLocal的缺点: 可能会造成内存泄漏,如果使用线程池的话,线程池中的线程不会被回收,线程中的ThreadLocalMap同样也不会被回收,这就可能导致内存泄漏 (memory leak)

    • 内存泄漏 memory leak : 申请的内存空间没有被正确释放,导致后续程序中,这块内存一直被占用
    • 内存溢出 out of memory: 存储的数据超出了指定空间的大小,导致数据越界

JVM中的共享区有哪些。 哪些可以作为gc root

  • 堆区和方法区是所有线程共享的,栈、本地方法栈、程序计数器是每个线程独有的
  • 首先了解,什么是gc root, 要了解gc root,需要先了解可达性分析算法,要了解可达性分析算法,需要先了解jvm 的gc策略
    • gc策略: jvm如何判断对象是否需要回收?
      1. 引用计数算法
        • 当一个对象创建,给对象中添加一个引用计数器,每当有一个地方引用此对象,计数器加1,反之减1,计数器为0的对象,就是可被回收的对象
      2. 可达性分析算法
        • 引用计数算法存在一个很明显的bug,就是循环引用, a对象引用b对象, b对象引用a对象,除此之外,没有其他地方引用a和b, 则a和b是可以被回收的,但是在引用计数算法中,它们都无法被回收
        • 可达性分析算法,基本思路是: 通过一系列被称为”gc root”的对象作为起点,从这个节点开始向下搜索,搜索走过的路径被称为引用链(Reference Chain), 当一个对象到gc root没有任何引用链相连(就是不可达),则认为此对象可以被回收.
  • 哪些可以作为gc root
    1. 虚拟机栈中引用的对象
    2. 方法区中类静态属性引用的对象
    3. 方法区中常量引用的对象
    4. 本地方法栈中JNI引用的对象 JNI(JAVA NATIVE INTERFACE 就是本地方法)

jvm内存模型

  1. 程序计数器
    • 线程私有,不会出现out of memory error ,主要作用为,在多线程环境下,记录线程执行的行号
  2. 虚拟机栈:
    • 线程私有,生命周期和线程一致,虚拟机栈是java方法执行的内存模型,每个方法在执行的同时都会创建一个栈帧,用于存储局部变量表,操作数栈,动态链接,方法出口等信息,局部变量表的内存空间在编译时完成分配,方法运行期间,不会改变局部变量表的内存空间大小;如果线程请求深度大于虚拟机栈所允许的深度,将抛出StackOverflowError;如果扩展时无法申请到足够的内存空间,将抛出OutOfMeeoryError
  3. 本地方法栈:
    • 和虚拟机栈类似,区别在于虚拟机栈执行的是java方法,而本地方法栈执行的是本地方法(native method),jvm并没有对native 方法的实现语言强制规定,所以可以自由实现;HotSpot虚拟机则直接将虚拟机栈和本地方法栈合二为一了;所以本地方法栈也会抛出StackOverflowError和OutOfMemoryError
  4. 堆
    • 所有线程共享,在虚拟机启动是创建;此内存区域的唯一目的就是存放对象实例,所有的对象和数组都要在堆上分配;java堆是gc管理的主要区域,也称为gc堆,因为gc采用分代收集算法,所以堆也可以分为新生代和老年代,更细一点还可以分为Eden空间,from Survivor,to Survivor等空间;另外堆中也可能划分出多个线程私有的分配缓冲区(Thread Local Alloocation Buffer,TLAB);如果堆中存在未分配完成的实例,并且堆内存又无法扩展,则会抛出OutOfMemoryError
  5. 方法区
    • 线程共享 ,用于存储已被虚拟机加载的类信息,常量,静态变量,即时编译器编译后的代码等数据,又被称为非堆(Non-Heap);在HotSopt虚拟机中,使用永久代来实现方法区 ,这样的话GC可以像管理堆内存一样管理这块内存区域;当方法区无法满足内存分配需求时，将抛出 OutOfMemoryError异常。

分布式锁的理解谈谈

分布式锁的本质就是,在分布式环境中,对共享资源进行互斥性访问

  • 基于Mysql,分布式环境中的线程连接同一个数据库,利用数据库中的行锁来达到互斥性访问,但是mysql的加锁和释放锁的效率比较低,不适用于实际生产环境
  • 基于Redis,通过使用SET EX NX命令,确保同一时间段内只有一个线程可能拿到锁,实现互斥性访问,
    • 详细介绍: redis分布式锁
  • 基于Zookeeper , 基于临时节点,一个客户端建立临时节点,获得锁,操作共享资源,操作完成后删除节点,释放锁,与redis不同的是,zookeeper不需要考虑锁过期的问题,客户端获取锁后,只要保持连接不断,就可以一直拥有锁,期间如果客户端崩溃,临时节点会被自动释放(因为zookeeper服务器与客户端维护了一个session来维持连接,而这个session依赖于客户端的定时心跳,如果zookeeper长时间收不到客户端的心跳,则自动删除节点)

redis集群策略

  • 主从模式: 主库可以读写,并且会和从库进行数据同步,这种模式下,客户端直接连接主库或者某一个从库,但是主库或是从库宕机的话,需要手动修改ip地址;并且这种模式比较难进行扩容,整个集群可存储的数据量受单台机器存储量的限制
  • 哨兵模式: 在主从模式的基础上,加入了哨兵节点,当主库宕机后,哨兵将会在从库中选出新的主库,另外哨兵也可以做集群,保证当某一哨兵节点宕机时,其他哨兵节点可以正常工作;哨兵模式可以保证redis集群的高可用,当仍不能解决redis集群容量上限的问题
  • Cluster模式: 这种模式会按照key进行槽位分配,可以使不同的key分配到不同的主节点上,使得redis集群可以支持更大的数据容量上限;每个主节点拥有多个从节点,,如果主节点宕机,则会选出选出新的主节点

实际生产环境中,根据实际需求来选择集群策略,如果有持续扩容需求的,选择Cluster模式;数据量不大的,可以选择哨兵模式

mysql中索引失效的情况

  • 没有符合最左前缀原则
    • mysql建立多列索引(联合索引)时,遵循最左前缀原则,即最左优先,例如 : create index xxx on t(a,b), 当查询条件是 b = ? 时,会遍历整个索引树
  • 字段进行了隐式数据类型转换
    • 在MySQL中，当操作符与不同类型的操作数一起使用时，会发生类型转换以使操作数兼容。则会发生隐式转换,例如 select 1 + ‘1’ 结果为 2 , select CONCAT(1,’1’) 的结果为 ‘11’字符串
  • ⾛索引没有全表扫描效率⾼

innodb如何实现事务

以update语句来举例

  • 当innodb收到一个update语句后,先根据条件查询到数据所在页,缓存到buffer pool中
  • 执行update语句,修改buffer pool中的数据
  • 针对此条update语句,生成一个RedoLog对象,存入buffer pool中
  • 针对此条update语句,生成undoLog日志
  • 如果事务提交,则将RedoLog对象进行持久化,将Buffer pool中的数据持久化到磁盘上
  • 如果事务回滚,则执行undoLog进行回滚

jdk1.7到jdk1.8 hashMap发生了哪些变化

  • 1.8 hashMap的底层是数组+链表+红黑树, 1.7的底层是数组+链表,加入红黑树主要是为了提高插入和查询效率
  • 1.7中put使用的是头插法,1.8中使用的是尾插法,因为需要判断元素个数,所以需要遍历链表,所以使用尾插法
  • 1.7中的hash算法比较复杂,因为需要提高散列性,提高hashmap的性能,而1.8因为红黑树的引入,可以适当简化hash算法,节省cpu资源

1.7到1.8 jvm发生了哪些变化

  • hotspot jvm 1.7中有永久代(j9,jRockit这些虚拟机都没有永久代), 1.8中没有,取而代之的是元空间, 元空间不使用虚拟机内存,而是使用本地内存,相比永久代,元空间的优势就是无需指定方法区大小,不会影响虚拟机内存

UDP和TCP的特点和区别

  1. 用户数据报协议 UDP (User Dategram Protocol)
    • 是无连接的,尽最大可能交付,没有拥塞控制,面向报文(对于应用程序传下来的报文不合并也不拆分,只是添加UDP部首),支持一对一,一对多,多对一和多对多的交互通信
  2. 传输控制协议 TCP (Transmission Control Protocol)
    • 是面向连接的,提供可靠交付,有流量控制,拥塞控制,提供全双工通信,面向字节流(把应用传下来的报文看成字节流,把字节流组织成大小不等的数据块),每一条TCP连接只能是点对点的(一对一)

tcp三次握手 四次挥手

握手

  • 客户端发送一个包含SYN标志的数据包到服务器
  • 服务器接受到数据包后，返回一个SYN/ACK应答包，在客户端未作出应答之前，服务器会将这些半连接放在一个缓存队列中
  • 客户端接受到服务器的应答包，在返回一个ACK确认包，完成TCP连接

挥手

  • 客户端发送一个 FIN 段，并包含一个希望接收者看到的自己当前的序列号 K. 同时还包含一个 ACK 表示确认对方最近一次发过来的数据。
  • 服务端将 K 值加 1 作为 ACK 序号值，表明收到了上一个包。这时上层的应用程序会被告知另一端发起了关闭操作，通常这将引起应用程序发起自己的关闭操作。
  • 服务端发起自己的 FIN 段，ACK=K+1, Seq=L。
  • 客户端确认。进入 TIME-WAIT 状态，等待 2 MSL（最大报文存活时间）后释放连接。ACK=L+1。

为什么要三次握手 而不是两次

tcp是双向通讯的协议，两次握手无法确保client接受到server的应答,也无法确保server接收到client的确认,

三次握手可以确保client和server都确认对方收到了自己的请求和响应,确保双方的收发功能都是正常的

消息队列确保消息可靠

两层含义, 1.如何控制消息不多 2.如何控制消息不少

  • 控制消息不多
    1. 消息的发送端不能不多发,较少出现,比较难控制
    2. 消息的消费端不能重复消费, 就是保证消息的幂等性(无论消费多少次,结果都是一样的,不会因为重复消费,而对结果产生影响),如何保证幂等性,看下一条问题
  • 控制消息不少,就是消息的丢失问题
    1. 生产者发送消息,要确认broker确实接受到了这条消息,比如RabbitMQ的confirm机制,kafka的ack机制
    2. broker要等待消费者真正确认消费了消息,才能删掉消息,就是消费端的ack机制,消费者消费完一条消息,给broker发送一个ack,确认消息消费无误,broker接收到ack后才会删除此条消息

MQ如何保证消息的幂等性

  • 先查再存 ,每次保存数据之前,都先去数据库查一下,如果存在就不存了,高并发下会有问题
  • 业务表添加唯一约束,针对不会分库分表的情况
  • 使用redis,针对分布式系统,又做了分库分表的,将消息id存在redis中,每次消费前去redis中查一下,存在就不消费了

并发编程三要素

  • 原子性 : 不可分割的操作,多个步骤要保证同时成功或同时失败
  • 有序性: 程序执行的顺序要和代码顺序一致
  • 可见性: 一个线程对共享变量的修改,其他线程可以马上看到

简述CAP

CAP理论：一个分布式系统最多只能同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition tolerance）这三项中的两项

  • C: consistency 指分布式系统中的数据一致性 all nodes see the same data at the same time
  • A: availability 指服务一直可用,并且是正常的响应时间 Reads and writes always succeed
  • P: Partition Tolerance 指分区容错性, 即分布式系统某一节点故障时,仍能提供满足一致性和可用性的服务 the system continues to operate despite arbitrary message loss or failure of part of the system

HTTPS 如何保证传输数据的安全性

  • 通过对称加密、非对称加密和数字证书

  • Client 发起一个HTTPS(https:/demo.linianhui.dev)的请求,根据RFC2818的规定，Client知道需要连接Server的443（默认）端口。

  • server将公钥证书(public key certificate)返回给client

  • Client验证公钥证书：比如是否在有效期内，证书的用途是不是匹配Client请求的站点，是不是在CRL吊销列表里面，它的上一级证书是否有效，这是一个递归的过程，直到验证到根证书（操作系统内置的Root证书或者Client内置的Root证书）。如果验证通过则继续，不通过则显示警告信息。

  • Client使用伪随机数生成器生成加密所使用的会话密钥，然后用server的公钥加密这个会话密钥，发给Server。

  • Server使用自己的私钥（private key）解密这个消息，得到会话密钥。至此，Client和Server双方都持有了相同的会话密钥。

  • Server使用会话密钥加密“明文内容A”，发送给Client。

  • Client使用会话密钥解密响应的密文，得到“明文内容A”。

  • Client再次发起HTTPS的请求，使用会话密钥加密请求的“明文内容B”，然后Server使用会话密钥解密密文，得到“明文内容B”。

  • 步骤1：客户端请求服务端的证书
    • 这一步被拦截、篡改了也没问题，因为只是通讯前的准备工作
  • 步骤2：服务端给客户端发送证书
    • 客户端收到证书后验真，如果OK，就有服务端的公钥了
    • 证书是公开的，被拦截了也没问题
    • 如果被篡改成假证书，并且客户端验证通过就不安全了
      • 客户端用假证书的公钥加密并发送，被拦截后可以被跟假公钥配对的私钥破解
      • 证书必须由专业结构（CA）发放
        • 发放机构对证书所有者审核
        • 如果使用假证书破解就可以根据家假证书找到破解者
  • 步骤3：客户端生成对称密钥并用公钥发送给服务端
    • 客户端记录下来生成的密钥
  • 步骤4：服务端解析出对称密钥并存储
  • 步骤5：客户端使用对称密钥加密数据发送
  • 步骤6：服务端使用对称密钥解密数据，并把响应数据用对称密钥加密

说说线程和进程

  • 进程是程序在操作系统中的一次执行过程,是系统运行程序的基本单位;进程是动态的,系统运行一个程序就是一个进程从创建到运行到消亡的过程; 在Java中,当我们启动Main函数时,其实就是启动了一个JVM进程,而Main函数所在的线程,被称之为主线程
  • 线程与进程类型,线程是一个比进程更小的执行单位;一个进程在其运行期间可以产生多个线程;线程和进程的最大不同在于,进程是各自独立的,而线程不是,同一进程中线程可能会相互影响.线程执行开销小,但不利于资源的管理和保护,而进程正好相反;在Java中,同一进程的多个线程之间,共享堆和方法区,而栈和程序计数器为每个线程私有;在进程中的切换线程工作的负担,比在操作系统中切换进程的负担小的多;所以,线程也被称之为轻量化进程

说一下线程私有的3块内存和私有的原因

  • 程序计数器

    1. 字节码解释器通过改变程序计数器来依次读取指令,从而实现代码的流程控制,比如:顺序执行、选择、循环、异常处理
    2. 在多线程的情况下,程序计数器用于记录当前线程的执行位置,从而当线程切换回来的时候,能够正确执行

    程序计数器私有主要是为了线程切换后能恢复到正确的执行位置

  • 虚拟机栈

    1. 每个java方法在执行的同时会创建一个栈帧用于存储局部变量表、操作数栈、常量池引用等信息.从方法调用直至执行完成的过程,就对应着一个栈帧在Java虚拟机中入栈和出栈的过程;

  • 本地方法栈

    1. 和虚拟机栈所发挥的作为非常相似,区别在于,虚拟机栈为虚拟机执行Java方法(就是字节码)服务,而本地方法栈则为虚拟机使用到的Native方法服务

    **为了保证线程中的局部变量不被其他线程访问到,虚拟机栈和本地方法栈是线程私有的 **

为什么要使用自增列作为主键

Innodb会使用自增列作为聚集索引（如果没有显示定义主键，innodb会选择第一个非null列作为主键，如果没有非null列，会使用隐式的ROWID）

数据记录是存主索引的叶子节点上的，所以要求同一叶子节点内的数据需要按照主键顺序排列

如果使用自增主键，则每次插入新数据，可以顺序添加到当前节点的后续位置；如果不使用，则每次插入新数据，就需要频繁的去移动、排序、读写，会增加很多开销，产生很多内存碎片，并且会得到不够紧凑的索引结构，后续需要通过OPTIMIZE TABLE来重建表优化填充页面

为什么使用索引可以提高效率

1.数据索引是有序的

2.在有序情况下，通过索引查询一个数据是无需遍历索引记录的

3.极端情况下，数据索引的查询效率为二分法查询效率，趋近于log2（N）

B数和B+树的区别

B树的每个节点都存储key和data，所有的节点组成这棵树，并且叶子节点指针为null，叶子节点不包含任何关键字信息

B+树的所有叶子节点包含所有的关键字信息和指向这个关键字信息对应数据的指针，并且叶子节点本身是依靠关键字的大小进行排序的；非叶子节点的都可以看作是索引的部分，非叶子节点中只含有其子树根节点中最大/最小的关键字（而B树的非叶子节点中也包含数据）

为什么说B+树比B树更适用于数据库索引

  1. B+树的磁盘读写开销低

  2. B+树的查询效率更稳定

快排

所有的未排序的部分,将第一个元素设为轴心点; 存储指数 = 轴心点指数 + 1 ; 然后遍历 存储指数到数组最右指数范围的数据,如果元素小于轴心点的元素,则交换当前元素和存储指数位置的元素,然后存储指数++ ;遍历完成后,交换轴心点元素和存储指数 - 1 位置的元素;

冒泡排序

依次比较相邻的两个数,将较小的数放在前面,较大的数放在后面,重复步骤直至排序完成,时间复杂度O(n^2) ,排序时间会随着数组元素数量增加而指数级增加,不适用于大数据量排序

计数排序

计数排序是一种非比较的排序算法,统计数组所有元素出现的次数,存入一个新的数组,遍历新数组,以数组下标作为值,下标对应的为数量, 这是一种空间换时间的思想,时间复杂度O(n+k)

设计模式

  • 单例模式:指一个类只有一个实例，且该类能自行创建这个实例的一种模式

    1. 单例类只有一个实例对象；
    2. 该单例对象必须由单例类自行创建；
    3. 单例类对外提供一个访问该单例的全局访问点。

  • 原型模式:用一个已经创建的实例作为原型，通过复制该原型对象来创建一个和原型相同或相似的新对象

  • 构造者模式:使用多个简单的对象一步一步构建成一个复杂的对象。这种类型的设计模式属于创建型模式，它提供了一种创建对象的最佳方式。

  • 工厂模式:这种类型的设计模式属于创建型模式，它提供了一种创建对象的最佳方式。在工厂模式中，我们在创建对象时不会对客户端暴露创建逻辑，并且是通过使用一个共同的接口来指向新创建的对象。

  • 代理模式:由于某些原因需要给某对象提供一个代理以控制对该对象的访问。这时，访问对象不适合或者不能直接引用目标对象，代理对象作为访问对象和目标对象之间的中介。

说说并发与并行的区别

  • 并发： 同一时间段，多个任务都在执行 (单位时间内不一定同时执行)；
  • 并行： 单位时间内，多个任务同时执行。

线程生命周期和状态

  • NEW 初始状态,刚被创建,还未调用start方法
  • RUNNABLE 运行中
  • BLOCKED 阻塞中
  • WAITING 等待状态
  • TIME_WAITING 超时等待,不同于WAITING,它可以在指定时间自行返回
  • TERMINATED 终止,执行完成

什么是上下文切换?

线程在执行过程中会有自己的运行条件和状态（也称上下文），比如上文所说到过的程序计数器，栈信息等。当出现如下情况的时候，线程会从占用 CPU 状态中退出。

  • 主动让出 CPU，比如调用了 sleep(), wait() 等。
  • 时间片用完，因为操作系统要防止一个线程或者进程长时间占用CPU导致其他线程或者进程饿死。
  • 调用了阻塞类型的系统中断，比如请求 IO，线程被阻塞。
  • 被终止或结束运行

这其中前三种都会发生线程切换，线程切换意味着需要保存当前线程的上下文，留待线程下次占用 CPU 的时候恢复现场。并加载下一个将要占用 CPU 的线程上下文。这就是所谓的 上下文切换。

上下文切换是现代操作系统的基本功能，因其每次需要保存信息恢复信息，这将会占用 CPU，内存等系统资源进行处理，也就意味着效率会有一定损耗，如果频繁切换就会造成整体效率低下。

什么是死锁,如何避免死锁

多个线程被同时阻塞,都在等在某一资源的释放;由于线程无限阻塞,导致应用无法正常终止

比如线程A持有锁1,线程B持有锁2,而它们同时都想申请对方的锁,这样就导致死锁

避免死锁: 1.线程结束后一定要释放锁 2. 线程申请锁按顺序申请,而释放锁的时候按反序释放,这样可以破坏循环等待的条件 3.线程申请锁如果失败,可以主动释放本身的锁

sleep和wait的区别

1.sleep不会释放锁 而wait会释放锁

2.sleep到时间会自动苏醒,wait需要别的线程进行唤醒

3.都是暂停线程执行

为什么我们调用 start() 方法时会执行 run() 方法，为什么我们不能直接调用 run() 方法?

因为调用 start() 方法方可启动线程并使线程进入就绪状态，直接执行 run() 方法的话不会以多线程的方式执行。

  • 继承Thread类,重写run方法
  • 实现Runnable接口,重写run方法
  • 实现Callable接口,重写call方法
